# uiux-audit — デモUI/UXの静的監査

「きれいかどうか」を感覚で判断せず、**CLAUDE.md §5-UI のチェックリストを1項目ずつ機械的に検査**するためのスキルです。
サーバ起動は禁止（CLAUDE.md §4）なので、すべて**コード上の静的チェック**で行います。

## 手順

### 0. 対象と性格を確定する

- 監査対象デモを確定（指定がなければ `AskUserQuestion` で確認。「全部」なら全デモを順に）。
- 対象デモの**種別**（Viteビルド型/静的バンドル型・CLAUDE.md §2）と**性格**（業務システム型/SaaS型・§2-2）を判定する。
  - README に性格の明記がなければ、それ自体を**指摘事項**として記録する（§2-2 で必須）。
- 静的バンドル型では、監査対象は `index.html`・オーバーレイスクリプト・`responsive.css` のみ。**`app.js` 内部は監査対象外**（無改変が正・修正提案もしない）。

### 1. 機械チェック（grep で拾えるもの）

対象デモのディレクトリで実行し、ヒットを記録する（Viteビルド型は `src/` と開発用 `index.html` を見る。生成済み `assets/*.js` はビルド産物なので文言チェックの対象外）:

```bash
# 仮置き文言の残留
grep -rin "lorem\|ipsum\|テストテスト\|sample text\|TODO\|FIXME" --include=*.tsx --include=*.ts --include=*.html --include=*.css <dir> | grep -v node_modules

# フォーカス殺し（代替スタイルなしの outline:none）
grep -rn "outline:\s*none\|outline: 0" --include=*.css --include=*.tsx <dir>

# 裸絶対パス（§8）
grep -rn 'href="/assets/\|src="/assets/' --include=*.html <dir>

# viewport / lang
grep -L 'name="viewport"' <dir>/index.html ; grep -L 'lang="ja"' <dir>/index.html

# 固定幅の疑い（375px超の width 直指定）
grep -rn "width:\s*[4-9][0-9][0-9]px\|width:\s*[1-9][0-9][0-9][0-9]px" --include=*.css --include=*.tsx <dir>
```

固定幅のヒットは即違反ではない。`max-width` 併用・スクロールラッパー内・`@media` 内なら OK。**前後を読んで判定**する。

### 2. 目視チェック（コードを読んで判定するもの）

§5-UI の残り項目を、該当コンポーネント/HTML を読んで判定する:

| 項目 | 見る場所 |
|------|----------|
| 日本語フォントスタック | ルートの `font-family` 定義 |
| デザイントークン運用（A型のみ） | `:root` の変数定義と、hex 直書きの散在 |
| テーブルの `overflow-x: auto` ラッパー | 一覧画面のテーブル周辺 |
| タップ対象 44px 以上 | ボタン・カードの `padding`/`height` |
| `:hover` / `:active` / `:focus-visible` | 主要インタラクティブ要素のCSS |
| 操作フィードバック（保存・送信後） | submit ハンドラ後の表示（トースト等） |
| 0件時の日本語文言 | 一覧・検索結果のレンダリング分岐 |
| `<label>` の有無・placeholder 依存 | フォーム項目 |
| 金額3桁区切り・日付 `YYYY/MM/DD`・数値右揃え | 明細・一覧のフォーマッタとCSS |
| 本文文字色のコントラスト | text系の色トークン（白背景に `#767676` より薄い本文は違反） |

### 2-2. 実システム体験チェック（§5-UX）

「本物のシステムを触っている」体験を壊す要素を、コードを読んで検出する:

| 項目 | 検出方法 |
|------|----------|
| 無反応UI（押しても何も起きないボタン・リンク） | 全ボタン/リンクに `onClick`/`href`/submit 配線があるかトレース。`onClick={() => {}}` や TODO ハンドラは違反 |
| `alert()`/`console.log` を操作の応答に使用 | `grep -rn "alert(\|console.log" src/ <dir>/*.js`（デバッグ残骸も違反） |
| 見せかけの検索・絞り込み・並び替え | 検索ボックス等の入力が実際にリストの絞り込みロジックに繋がっているか |
| 固定値の合計・追従しない計算 | 数量・単価の変更が小計・税・合計の再計算に反映されるか（計算ロジックの有無） |
| 操作が state に反映されない | 作成・編集・削除のハンドラが state 更新＋完了フィードバックまで行うか |
| 実演フローの行き止まり | README の実演フローに沿って画面遷移をトレースし、進めない/戻れない画面がないか |
| 空っぽの初期画面・利用者コンテキスト欠如 | 初期データの件数・ステータス分布、ヘッダーの担当者名表示 |

### 3. 性格別チェック（§2-2）

- **業務システム型**: `<title>`・画面見出し・メニュー・ボタン・エラー/空状態文言に英語が混ざっていないか。
  ```bash
  grep -o "<title>[^<]*</title>" <dir>/index.html
  grep -rn ">\s*[A-Za-z][A-Za-z ]\{3,\}\s*<" --include=*.tsx --include=*.html <dir>   # 英語ラベルの疑い（コード値・型番は除外して判定）
  ```
- **SaaS型**: プロダクト名が架空か・実在SaaSと紛らわしくないか（怪しければ CLAUDE.md §11 に従い `AskUserQuestion`）。

### 4. 報告（このスキルの成果物）

**勝手に直さない。** 監査結果を次の形式で報告して終える（修正は指示があれば `fix-demo` スキルで行う）:

```
## uiux-audit 結果: <demo-directory>（種別: A/B・性格: 業務システム型/SaaS型）

### ❌ 違反（要修正）
- <ファイル>:<行> — <§5-UI / §5-UX のどの項目に違反か> — <1行の修正方針>

### ⚠️ 要判断（ユーザー確認推奨）
- ...

### ✅ 合格した項目
- （項目名の列挙でよい）
```

- 違反ゼロなら「§5-UI 全項目合格」と明記する。
- 修正まで依頼されている場合は、報告後に `fix-demo` の手順（種別別の修正 → ビルド/整合 → preflight → コミット案）へ進む。静的バンドル型で違反が `app.js` 由来（描画される文言がバンドル内）の場合、修正は**オーバーレイでの上書き**になる。影響が大きければ §11 で確認する。

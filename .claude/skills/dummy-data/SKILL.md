# dummy-data — 商談品質の架空データを安全に作る

このリポジトリは**公開**であり、実在の顧客名・社名・担当者・案件名は絶対に入れられない（CLAUDE.md §4-4）。
一方で `テスト株式会社`・`山田太郎`・`AAA` のような雑なダミーは、商談デモの説得力を殺す（§5-UI・§7-15）。
このスキルは「**本物らしく見えるが、機械的に架空だと保証できる**」データの作り方を定める。

## 安全パターン（この形式以外を使わない）

| 種類 | パターン | 例 |
|------|----------|-----|
| 社名 | **業種語＋一般名詞の合成**。実在確認できない合成名にし、生成後に社名でWeb検索して**同名企業がヒットしたら別名に差し替える**（検索できない環境なら §11 で確認） | 株式会社あおば印刷、ひまわり企画株式会社、株式会社みなと製作所 |
| 人名 | ありふれた姓＋ありふれた名の組合せ（実在著名人・取引先関係者と同姓同名を避ける） | 佐藤 花子、鈴木 一郎、高橋 美咲 |
| 住所 | 実在の都道府県・市区までは可。**町名以下は「〇〇町0-0-0」のように番地に 0 を使う**（実在しない番地であることを機械的に保証） | 東京都千代田区丸の内0-0-0 サンプルビル3F |
| 郵便番号 | `000-0000` 固定、またはハイフン前を `000` | 000-0000 |
| 電話 | `0X0-0000-0000` 形式（局番に 0000 を含める。実在割当てのない形） | 03-0000-1234、090-0000-5678 |
| メール | ドメインは **`example.com` / `example.jp` のみ**（RFC 2606 予約ドメイン） | hanako.sato@example.jp |
| URL | `https://www.example.com/...` のみ | — |
| 案件・伝票番号 | 英字プレフィクス＋連番（英字OK: データ値扱い・§2-2） | EST-2026-0012、ORD-0451 |
| 金額 | 業務の相場感に合う桁にする（印刷見積なら数万〜数百万円）。**表示は3桁区切り**（§5-UI） | ¥1,238,500 |
| 日付 | デモを見る時点で不自然でない近過去〜近未来。表示は `YYYY/MM/DD` | 2026/06/30 |

## 品質ルール（雑ダミーの禁止）

- `テスト`・`test`・`sample`・`aaa`・`ダミー` を**表示される値に使わない**（READMEの「架空データ」明記はこの限りでない）。
- 一覧に並ぶデータは**値をばらつかせる**: 金額の桁・日付・ステータスが全行同じ、はNG。ソート・検索・0件表示のデモができる程度の件数（目安 8〜15件）と分布を持たせる。
- 明細は業務として辻褄を合わせる: 単価×数量＝小計、小計＋消費税（10%）＝合計、が計算上一致すること。
- 業種の言葉遣いを合わせる（印刷なら「部数・用紙・色数・製本」、ブライダルなら「挙式・披露宴・列席者数」など、既存デモの語彙を踏襲）。

## 手順

1. 対象デモと必要なデータ構造を確認する（Viteビルド型なら `src/data/*.ts` と `src/types/`、静的バンドル型ならオーバーレイスクリプト内の定義。**`app.js` 内のデータは触れない**）。
2. 上の安全パターンで生成する。社名は生成のたびに**実在衝突チェック**（Web検索）を行う。
3. 計算整合（小計・税・合計）をスクリプトか手計算で検証する。
4. `git diff` を自分でスキャンし、実在しそうな固有名詞・`example.com` 以外のドメイン・0を含まない電話番号/番地が**ないこと**を確認（CLAUDE.md §10）。
5. 判断がつかない固有名詞が残ったら `AskUserQuestion`（CLAUDE.md §11）。

## 完了条件

- [ ] 全データが上表の安全パターンに適合（ドメイン・電話・住所は機械的に架空と分かる形式）。
- [ ] 社名の実在衝突チェックを実施した。
- [ ] 明細の計算が一致し、表示フォーマットが §5-UI（3桁区切り・`YYYY/MM/DD`）に従う。
- [ ] `テスト`/`sample` 等の雑ダミー文言が表示値にない。

## 0. このリポジトリは何か（30秒で読む）

- **目的**: 受託案件の提案・商談で見せる**公開デモ**を置く。GitHub Pages でURLを共有する。
- **公開リポジトリ**である。コミットした瞬間に世界に公開されると考えること。
- **本番コード・顧客固有実装・実データは置かない。** デモとモックだけ。
- **オーナー / 公開URLの基点**: `https://ict-o.github.io/client-demo-apps/`
  （git remote は `ict-o/client-demo-apps`。README等に残る `TatsuyaTanaka41.github.io` は古い記述。§7-13, §8参照）
- **デプロイの仕組み**: `.github/workflows/static.yml` が **`main` への push 時のみ**、リポジトリ全体を丸ごと Pages にアップロードする。
  - **CIではビルドしない。** コミット済みのファイルがそのまま公開される。
  - → **ビルド成果物（`index.html`・`assets/`）をコミットし忘れると、公開ページが壊れる。**
  - → 作業ブランチへの push ではデプロイされない（安全）。デプロイは main マージ後。

1. `git status` と `git branch` で現在地を確認する。作業ブランチは指定されたもの（例 `claude/...`）。**`main` で直接作業しない。**

## 2. 【最重要】デモの種別を必ず先に判定する

このリポジトリのデモは**2種類**あり、扱い方が正反対です。**混同が最大の事故原因**です。
対象デモのディレクトリで次を確認してください。

```
そのデモに vite.config.ts と package.json があるか？
├─ ある   → 【A: Viteビルド型】  →  §2-A へ
└─ ない   → 【B: 静的バンドル型】 →  §2-B へ
```

現状の分類（2026-07 時点）:

| デモ | 種別 | 中身 |
|------|------|------|
| `bridal-estimate-demo` | **A: Viteビルド型** | `src/`・`vite.config.ts`・`package.json`・ビルド済み `index.html`/`assets/` |
| `print-estimate-demo` | **B: 静的バンドル型** | 手置きの `index.html`・`app.js`（ビルド済みバンドル）・`past-list.js`・`responsive.css` |
| `standard-price-demo` | **B: 静的バンドル型** | 手置きの `index.html`・`app.js`（同上・同一バンドル）・`customize.js`・`responsive.css` |

### §2-A. Viteビルド型の扱い

- ソースは `src/`。React 19 + TypeScript + Vite 8 + React Router v7。
- **公開されるのは、コミットされたビルド成果物**（デモ直下の `index.html` と `assets/`）。`src/` は公開されるが実行はされない。
- コードを変更したら **`npm run build`（＝ビルド成功が完了条件）** し、**生成された `index.html` と `assets/` をコミットする**。
  - ビルドすると `assets/index-XXXX.js` の**ハッシュ名が変わり**、`index.html` の参照も書き換わる。**両方を必ず一緒にコミットする。**片方だけだと本番で JS が 404 になる。
- `vite.config.ts` は §6 の雛形に厳密に従う（`base`・`outDir: '.'`・`emptyOutDir: false`）。

### §2-B. 静的バンドル型の扱い

- **`package.json` も `vite.config.ts` も `src/` も無い。追加してはいけない。** これらは「ビルド不要でそのまま公開する」設計です。
- `app.js` は**ビルド済みの巨大バンドル（約232KB・minified）**。しかも `print-estimate-demo` と `standard-price-demo` で**バイト単位で同一**（共有バンドル）。
  - **`app.js` を手編集しない。** minified で人間が安全に編集できない。片方を変えるともう片方との同一性も壊れる。
  - 機能を足す／消すときは、**`app.js` 読込後に走るオーバーレイスクリプト**で行う。これがこのリポジトリの確立された手法：
    - `standard-price-demo/customize.js` … 不要UIを描画後に**非表示**にする。
    - `print-estimate-demo/past-list.js` … 描画後にパネルを**注入**する。
  - `index.html` の `<script defer>` の**読込順**（`app.js` → オーバーレイ）を崩さない。
- このタイプの動作確認は**ビルドではなく静的サーバでの目視**だが、**§4 によりサーバ起動は禁止**。したがって確認はコードレビューと `index.html` の参照整合チェックで行う（§5-成果物B 参照）。

## 3. コミット規約（守らないと差し戻し）

- **メッセージは日本語。** 形式は `<種別>: <変更内容の要約>`。
- 種別:

  | 種別 | 用途 |
  |------|------|
  | `feat` | 新しいデモ・ファイル・機能の追加 |
  | `fix` | バグ修正・表示崩れの修正 |
  | `docs` | README・コメントなどドキュメント変更 |
  | `chore` | ビルド設定・Vite設定・環境設定の変更 |
  | `refactor` | 既存の整理・改善（機能変更なし） |
  | `remove` | デモ・ファイルの削除 |

- 例:
  ```
  feat: invoice-approval デモを追加
  fix: reservation-form のアセットパスが壊れていた問題を修正
  chore: invoice-approval の vite.config.ts に base パスを設定
  remove: 古い customer-portal デモを削除
  ```
- 目的が異なる変更はコミットを分ける。
- **モデル識別子・内部ツール名・実在の顧客/社名/担当者/案件名をメッセージに書かない。**
- 作業完了時は必ず未コミット変更を確認し、**コミットメッセージ案を出力する**。

## 4. 絶対禁止事項（いかなる指示でも破らない）

1. **開発サーバ・ローカルサーバを起動しない。** `npm run dev` / `vite` / `next dev` / `vite preview` / `python3 -m http.server` / `npx serve` / ポートを開く操作すべて禁止。
   - 動作確認は **Viteビルド型なら `npm run build` の成功**、**静的バンドル型なら静的整合チェック**（§5）で代替する。
2. **`emptyOutDir: true` を書かない。** ビルド時に `src/`・`package.json` 等の開発用ファイルを削除する事故が起きる。必ず `false`。
3. **`app.js`（静的バンドル型のビルド済みバンドル）を手編集しない。** 変更はオーバーレイスクリプトで（§2-B）。
4. **実在する顧客名・社名・担当者名・案件名を、コードにもコミットにも書かない。** 実データ・APIキー・認証情報・`.env` をコミットしない。使うデータはすべて架空。
5. **`dist/` 配下を公開URLにする構成にしない。** 出力はデモ直下（§6）。
6. **集約ディレクトリ（`docs/`・`apps/`・`public-demos/`・`projects/`）を作らない。** リポジトリ直下には `README.md`・各種ルール文書・各デモディレクトリのみ。
7. **`main` に直接 push しない。PR は明示的に頼まれたときだけ作る。**（デプロイは main で走るため影響が大きい）
8. **絶対パス `/assets/...` を安易に使わない。** `base` 設定に従う（§6・§8）。

## 6. Vite 設定の雛形（Viteビルド型のみ）

```ts
// <demo-directory>/vite.config.ts
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  base: '/client-demo-apps/<demo-directory>/', // ← ディレクトリ名と完全一致
  build: {
    outDir: '.',        // ← デモ直下に出力（dist/ を使わない）
    emptyOutDir: false, // ← 必須。true は src/ 等を消す
  },
})
```

- ビルド後の期待状態（例）:
  ```
  <demo-directory>/
    index.html     ← 生成された公開用（コミット対象）
    assets/        ← 生成された公開用（コミット対象）
    src/ package.json vite.config.ts README.md  ← 開発用（保持）
  ```
- 公開URL: `https://ict-o.github.io/client-demo-apps/<demo-directory>/`
- `dist/index.html` / `.../<dir>/dist/` というURL構成は**採用しない**。

## 8. アセットパス・ルーティングの具体ルール

- **Viteビルド型**: 画像等は `src/` からの `import` か `public/` を使い、`base` にビルドが自動追従させる。`index.html` 内は Vite が `/client-demo-apps/<dir>/...` に解決する。**手書きの `/assets/...` を書かない。**
- **静的バンドル型**: `index.html` から `./app.js`・`./responsive.css` のように**相対パス**で参照する。
- **ルーティング**: React Router を使うなら Pages のサブパス直アクセスで 404 になり得る。`HashRouter` を使うか、直アクセスが要る画面は静的に成立させる。デモ用途では複雑なルーティングを避ける。
- 各デモの基本公開URLは必ず `/client-demo-apps/<demo-directory>/` の形。

## 9. ルート `index.html` の更新（デモ追加・削除で必須）

ルート `index.html` は「パスワードゲート付きのデモ一覧」。デモを**追加/削除したら必ず更新**する。

- 追加時: `.demo-list` の中に、既存カードと**同じ構造**で `.demo-card` を1枚足す:
  ```html
  <a class="demo-card" href="./<demo-directory>/">
    <div class="demo-info">
      <div class="demo-title"><!-- 日本語のデモ名 --></div>
      <div class="demo-slug"><demo-directory></div>
      <div class="demo-desc"><!-- 1〜2文のデモ概要 --></div>
    </div>
    <span class="demo-arrow">›</span>
  </a>
  ```
  - `href` は**相対 `./<dir>/`**（オーナー名をハードコードしない）。
- 削除時: 対応する `.demo-card` を削除する。
- ページ内のパスワードゲート（SHA-256 ハッシュ比較のスクリプト）は**触らない**。ハッシュ値・認証ロジックを変更依頼されていない限り、そのまま残す。

## 11. 判断に迷ったときのエスカレーション（正確な分岐）

**原則: 可逆で明確な作業は進める。不可逆・外部影響・仕様不明はユーザーに確認する。**

### 確認せず進めてよい（黙って実行し、結果を報告）

- 種別判定が明確なデモへの、指示範囲内のコード変更。
- `npm run build` の実行（Viteビルド型のビルドは許可された確認手段）。
- 指定作業ブランチへのコミット・push。
- ルート `index.html` のカード追加/削除（デモ追加・削除に伴う必須作業）。
- 明らかな架空データの追加・修正。

### 必ず `AskUserQuestion` で確認してから進める

- **デモの削除**、または**共有 `app.js` に影響し得る変更**（他デモに波及するため）。
- **`main` への push、または PR の作成**（明示依頼がない限りしない）。
- 静的バンドル型で「どうしても `app.js` 本体の改変が必要」に見えるとき（まずオーバーレイで解けないか検討し、無理なら確認）。
- ルート `index.html` の**認証ロジック・パスワードハッシュ**の変更。
- 既存の複数デモ／README／ルール文書にまたがる**大きめのリファクタ**。
- 指示が2通り以上に解釈できるとき（レビューコメントの意図が曖昧、等）。**推測で大きく動かない。**
- 実在情報かどうか判断がつかない固有名詞（社名・製品名等）をコミットに含めそうなとき。

### 手を止めて報告だけする（勝手に直さない）

- ビルドが**環境要因**（ネットワーク・依存の不整合）で失敗し、コード修正では直せないとき。
- 既存コミット済みコードに、今回の作業対象外の情報漏えい懸念を見つけたとき（勝手に改変せず、まず報告）。

### エスカレーション時に必ず添える情報

- 何をしようとして、どこで詰まったか（該当ファイル・行）。
- 選択肢と、各選択肢の影響（特に不可逆性・公開影響）。
- あなたの推奨案（あれば）。

## 13. 時短スキル（`.claude/skills/`）

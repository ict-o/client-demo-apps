## 0. このリポジトリは何か（30秒で読む）

- **目的**: 受託案件の提案・商談で見せる**公開デモ**を置く。GitHub Pages でURLを共有する。
- **公開リポジトリ**である。コミットした瞬間に世界に公開されると考えること。
- **本番コード・顧客固有実装・実データは置かない。** デモとモックだけ。
- **オーナー / 公開URLの基点**: `https://ict-o.github.io/client-demo-apps/`
  （git remote は `ict-o/client-demo-apps`。README等に残る `TatsuyaTanaka41.github.io` は古い記述。§7-13, §8参照）
- **デプロイの仕組み**: `.github/workflows/static.yml` が **`main` への push 時のみ**、リポジトリ全体を丸ごと Pages にアップロードする。
  - **CIではビルドしない。** コミット済みのファイルがそのまま公開される。
  - → **ビルド成果物（`index.html`・`assets/`）をコミットし忘れると、公開ページが壊れる。**
  - → 作業ブランチへの push ではデプロイされない（安全）。デプロイは main マージ後。

## 2. 【最重要】デモの種別を必ず先に判定する

### §2-0. 新規デモの標準方針（新しく作るときの既定）

**新しいデモは、原則 A: Viteビルド型（最新技術スタック）で作る。**

- スタック: **React 19 + TypeScript + Vite（最新安定版）**。ルーティングが要るなら React Router v7。
- **ビルド時に静的ファイルを生成し、GitHub Pages でそのまま閲覧できる構成にする**（サーバ処理・バックエンド前提にしない）。
  - `npm run build` → デモ直下に `index.html` と `assets/`（静的ファイル）を生成 → コミット → main マージで Pages 公開。
- バージョンは**その時点の最新安定版**を採用する。`package.json` を古い値で固定コピーせず、新規作成時に最新へ更新する（例: React・Vite・TypeScript のメジャーは最新安定を選ぶ）。
- **B: 静的バンドル型は新規作成しない。** 既存の共有 `app.js` を流用して派生を作る場合（例 `standard-price-demo`）に限り §2-B / `clone-demo` スキルを使う。ゼロから新規デモを B 型で作らない。

> つまり: **ゼロから作る＝A型（最新React+Vite、ビルドで静的生成）。既存バンドルの派生＝B型（clone-demo）。**

### §2-1. 既存デモを触るときは種別を判定する

既存デモは**2種類**あり、扱い方が正反対です。**混同が最大の事故原因**です。
対象デモのディレクトリで次を確認してください。

```
そのデモに vite.config.ts と package.json があるか？
├─ ある   → 【A: Viteビルド型】  →  §2-A へ
└─ ない   → 【B: 静的バンドル型】 →  §2-B へ
```

現状の分類（2026-07 時点）:

| デモ | 種別 | 中身 |
|------|------|------|
| `bridal-estimate-demo` | **A: Viteビルド型** | `src/`・`vite.config.ts`・`package.json`・ビルド済み `index.html`/`assets/` |
| `print-estimate-demo` | **B: 静的バンドル型** | 手置きの `index.html`・`app.js`（ビルド済みバンドル）・`past-list.js`・`responsive.css` |
| `standard-price-demo` | **B: 静的バンドル型** | 手置きの `index.html`・`app.js`（同上・同一バンドル）・`customize.js`・`responsive.css` |

### §2-A. Viteビルド型の扱い

- ソースは `src/`。React 19 + TypeScript + Vite 8 + React Router v7。
- **公開されるのは、コミットされたビルド成果物**（デモ直下の `index.html` と `assets/`）。`src/` は公開されるが実行はされない。
- コードを変更したら **`npm run build`（＝ビルド成功が完了条件）** し、**生成された `index.html` と `assets/` をコミットする**。
  - ビルドすると `assets/index-XXXX.js` の**ハッシュ名が変わり**、`index.html` の参照も書き換わる。**両方を必ず一緒にコミットする。**片方だけだと本番で JS が 404 になる。
- `vite.config.ts` は §6 の雛形に厳密に従う（`base`・`outDir: '.'`・`emptyOutDir: false`）。

### §2-B. 静的バンドル型の扱い

- **`package.json` も `vite.config.ts` も `src/` も無い。追加してはいけない。** これらは「ビルド不要でそのまま公開する」設計です。
- `app.js` は**ビルド済みの巨大バンドル（約232KB・minified）**。しかも `print-estimate-demo` と `standard-price-demo` で**バイト単位で同一**（共有バンドル）。
  - **`app.js` を手編集しない。** minified で人間が安全に編集できない。片方を変えるともう片方との同一性も壊れる。
  - 機能を足す／消すときは、**`app.js` 読込後に走るオーバーレイスクリプト**で行う。これがこのリポジトリの確立された手法：
    - `standard-price-demo/customize.js` … 不要UIを描画後に**非表示**にする。
    - `print-estimate-demo/past-list.js` … 描画後にパネルを**注入**する。
  - `index.html` の `<script defer>` の**読込順**（`app.js` → オーバーレイ）を崩さない。
- このタイプの動作確認は**ビルドではなく静的サーバでの目視**だが、**§4 によりサーバ起動は禁止**。したがって確認はコードレビューと `index.html` の参照整合チェックで行う（§5-成果物B 参照）。

### §2-2. 【必須】デモの性格を宣言する: 業務システム型 / SaaS型

技術種別（A/B）とは**別の軸**として、すべてのデモは次のどちらかに分類し、**デモの README に「性格: 業務システム型」または「性格: SaaS型」を明記**する。この分類で UI の文言ルール（§5-UI）が変わる。

**判定基準**:

```
このデモは「提案先1社の社内業務（見積・受発注・在庫・承認・帳票…）を
置き換える/補助するシステム」の提案か？
├─ はい → 【業務システム型】（このリポジトリの主用途。迷ったらこちら）
└─ いいえ（不特定多数に売るプロダクトのモック） → 【SaaS型】
```

**業務システム型のルール**（商談相手は非IT部門の担当者・経営者。英語UIは「難しそう」という第一印象を与え、商談で不利になる）:

- **システム名・画面名・メニュー・ボタン・見出し・エラーメッセージ・空状態文言まで、すべて日本語。**
  - システム名を英語にしない（× `Bridal Estimate` → ○「ブライダル見積作成システム」）。
  - 英字を使ってよいのは、データ値としてのコード・型番・品番等（例 `EST-2026-001`）のみ。
- `<title>` は日本語で「<画面名またはデモ名> | <システム名>」の形式。
- 一覧は**テーブル中心**、金額・数量セルは**右揃え**。操作ボタンは文字ラベル付き（アイコンだけのボタンにしない。どうしても使うなら `aria-label` を付ける）。

**SaaS型のルール**:

- プロダクト名は英語でもよいが**架空名に限る**（実在SaaSと紛らわしい名前は §4-4 に抵触。命名前に軽く検索して衝突がないか確認）。
- UI文言は日本語基調（商談相手は国内企業）。英語はプロダクト名・ロゴ・キャッチコピー程度に留める。
- LP的トップページ・オンボーディング風の導線を持たせてよい。

**現状の分類（2026-07 時点）**: `bridal-estimate-demo`・`print-estimate-demo`・`standard-price-demo` は**すべて業務システム型**。

> 既知の逸脱: `bridal-estimate-demo` の開発用 `index.html` の `<title>` が英語（`Bridal Estimate`）のまま。次にこのデモを触るとき（リビルドが発生するとき）に日本語へ修正し、README の旧URL（`TatsuyaTanaka41`）も同時に直すこと。**この修正だけのために勝手にリビルドしない。**

## 3. コミット規約（守らないと差し戻し）

- **メッセージは日本語。** 形式は `<種別>: <変更内容の要約>`。
- 種別:

  | 種別 | 用途 |
  |------|------|
  | `feat` | 新しいデモ・ファイル・機能の追加 |
  | `fix` | バグ修正・表示崩れの修正 |
  | `docs` | README・コメントなどドキュメント変更 |
  | `chore` | ビルド設定・Vite設定・環境設定の変更 |
  | `refactor` | 既存の整理・改善（機能変更なし） |
  | `remove` | デモ・ファイルの削除 |

- 例:
  ```
  feat: invoice-approval デモを追加
  fix: reservation-form のアセットパスが壊れていた問題を修正
  chore: invoice-approval の vite.config.ts に base パスを設定
  remove: 古い customer-portal デモを削除
  ```
- 目的が異なる変更はコミットを分ける。
- **モデル識別子・内部ツール名・実在の顧客/社名/担当者/案件名をメッセージに書かない。**
- 作業完了時は必ず未コミット変更を確認し、**コミットメッセージ案を出力する**。

## 4. 絶対禁止事項（いかなる指示でも破らない）

1. **開発サーバ・ローカルサーバを起動しない。** `npm run dev` / `vite` / `next dev` / `vite preview` / `python3 -m http.server` / `npx serve` / ポートを開く操作すべて禁止。
   - 動作確認は **Viteビルド型なら `npm run build` の成功**、**静的バンドル型なら静的整合チェック**（§5）で代替する。
2. **`emptyOutDir: true` を書かない。** ビルド時に `src/`・`package.json` 等の開発用ファイルを削除する事故が起きる。必ず `false`。
3. **`app.js`（静的バンドル型のビルド済みバンドル）を手編集しない。** 変更はオーバーレイスクリプトで（§2-B）。
4. **実在する顧客名・社名・担当者名・案件名を、コードにもコミットにも書かない。** 実データ・APIキー・認証情報・`.env` をコミットしない。使うデータはすべて架空。
5. **`dist/` 配下を公開URLにする構成にしない。** 出力はデモ直下（§6）。
6. **集約ディレクトリ（`docs/`・`apps/`・`public-demos/`・`projects/`）を作らない。** リポジトリ直下には `README.md`・各種ルール文書・各デモディレクトリのみ。
7. **`main` に直接 push しない。PR は明示的に頼まれたときだけ作る。**（デプロイは main で走るため影響が大きい）
8. **絶対パス `/assets/...` を安易に使わない。** `base` 設定に従う（§6・§8）。

### §5-UI. 共通UI/UX合格条件（画面を作った/変えた全成果物に適用）

「見た目をきれいに」ではなく、以下を**1項目ずつ機械的に確認**する。サーバは起動できない（§4）ので、確認方法は**コード上のチェック**として書いてある。

**構造・基本**:

- [ ] `<html lang="ja">` と `<meta name="viewport" content="width=device-width, initial-scale=1.0">` がある。
- [ ] フォント指定が日本語対応スタック（`Hiragino Sans` / `Noto Sans JP` 系。ルート `index.html`・`bridal-estimate-demo/src/index.css` と同系統）。
- [ ] Viteビルド型では、色・角丸・影・フォントを **`:root` の CSSカスタムプロパティ（デザイントークン）** で定義し、コンポーネント側は `var(--...)` を参照する（`bridal-estimate-demo/src/index.css` が手本）。同一デモ内に同じ用途の hex 直書きを散らかさない。

**レスポンシブ**（商談はスマホ・タブレットで見せることがある）:

- [ ] コンテナ幅は `max-width`＋`%`/`flex`/`grid` で組み、**375px 幅で横スクロールが出る固定 `width: NNNpx` 指定がない**。
- [ ] 幅の広いテーブルは `overflow-x: auto` のラッパーで包む（ページ全体を横スクロールさせない）。
- [ ] タップ対象（ボタン・カード・リンク行）の実効高さが **44px 以上**（padding 込みで確認）。

**インタラクション状態**（「押せるのか分からない」「押したのに反応がない」を潰す）:

- [ ] 主要なボタン・カード・リンクに `:hover` と `:active` のスタイルがある（ルート `index.html` の `.demo-card` が手本）。
- [ ] `outline: none` を**代替のフォーカススタイルなしで**書いていない（`:focus-visible` で枠か影を出す）。
- [ ] 保存・送信・検索などの操作に**結果のフィードバック**（トースト・完了表示・結果件数など、日本語文言）がある。

**文言・データ表示**:

- [ ] 一覧・検索結果に **0件時の日本語文言**（例「該当する見積はありません」）が実装されている。
- [ ] フォームの必須項目に**日本語のバリデーションメッセージ**があり、`placeholder` をラベル代わりにしていない（`<label>` を付ける）。
- [ ] 金額は `¥1,234,567` または `1,234,567円` の**3桁区切り**、日付は `YYYY/MM/DD`、金額・数量のテーブルセルは右揃え。
- [ ] `Lorem ipsum`・`テストテスト`・`TODO`・`sample text` 等の仮置き文言が残っていない（`grep -rin "lorem\|TODO" <dir>` で確認）。
- [ ] 本文テキストの文字色が薄すぎない（白背景に対し `#767676` より薄いグレーを本文に使わない。補足テキストは既存トークン `--text-sub` 相当まで）。

**性格別（§2-2）**:

- [ ] 業務システム型: UI文言・システム名・`<title>` が**すべて日本語**（英字はコード等のデータ値のみ）。
- [ ] SaaS型: プロダクト名が架空で、実在SaaSと紛らわしくない。

### §5-UX. 実システム体験合格条件（新規デモ・画面追加に適用）

デモの目的は「**本物のシステムを操作している**と商談相手に感じさせる」こと。静止画のようなモック・飾りだけのUIは不合格。以下を**1項目ずつ確認**する。

**「押せるのに動かない」を全滅させる**（実システム感を最も壊すのは無反応UI）:

- [ ] 画面上の**すべてのボタン・リンク・メニュー項目が、押すと必ず何かが起きる**（画面遷移・モーダル・トースト・状態変化のいずれか）。1つでも無反応の要素があれば不合格。
- [ ] デモで実装しない機能は、(a) そもそも置かない、または (b) `disabled` 見た目＋日本語ツールチップ/注記にする。**押せる見た目で無反応**にしない。
- [ ] `alert()`・`console.log` を操作の応答に使わない（モーダル・トーストコンポーネントで返す）。

**操作が本当に反映される**（見せかけの入力フォームにしない）:

- [ ] 作成・編集・削除・ステータス変更が**画面上の state に即時反映**される（保存→一覧に新しい行が現れる、編集→値が変わる、削除→行が消える＋日本語の完了フィードバック）。リロードで消えてよい（永続化は不要。`useState`/メモリ内でよい）。
- [ ] 入力値が計算・集計に**実際に効く**（数量を変えると小計・税・合計が即時再計算される等）。表示だけの電卓・固定値の合計にしない。
- [ ] 検索・絞り込み・並び替えのUIを置いたなら**実際に機能する**（飾りの検索ボックス禁止。機能させないなら置かない）。

**業務フローが最後まで通る**:

- [ ] READMEの「実演フロー」に書いた一連の操作（例: 一覧 → 新規作成 → 明細編集 → プレビュー → 出力）が、**途切れずに最後まで画面上で完走できる**。行き止まり画面（戻れない・次へ進めない）がない。
- [ ] 初期表示の時点で**業務が進行中に見える**データが入っている（一覧に現実的な件数・多様なステータス・直近の日付。空のシステムを見せない。`dummy-data` スキル参照）。
- [ ] ヘッダー等に**利用者コンテキスト**（例: ログイン中の担当者名・部署名。架空名）が表示され、「誰かが使っている業務画面」に見える。

**確認方法**（サーバ起動禁止・§4 のため）: Viteビルド型はコード上で、全 `onClick`/`onSubmit` にハンドラが配線されているか・stateが更新されるか・行き止まりの画面がないかをコンポーネントを読んで確認する。`grep -rn "onClick" src/ | grep -v "=>"` 等で未配線を探し、README の実演フローに沿って遷移をコード上でトレースする。

## 6. Vite 設定の雛形（Viteビルド型のみ）

```ts
// <demo-directory>/vite.config.ts
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  base: '/client-demo-apps/<demo-directory>/', // ← ディレクトリ名と完全一致
  build: {
    outDir: '.',        // ← デモ直下に出力（dist/ を使わない）
    emptyOutDir: false, // ← 必須。true は src/ 等を消す
  },
})
```

- ビルド後の期待状態（例）:
  ```
  <demo-directory>/
    index.html     ← 生成された公開用（コミット対象）
    assets/        ← 生成された公開用（コミット対象）
    src/ package.json vite.config.ts README.md  ← 開発用（保持）
  ```
- 公開URL: `https://ict-o.github.io/client-demo-apps/<demo-directory>/`
- `dist/index.html` / `.../<dir>/dist/` というURL構成は**採用しない**。

## 8. アセットパス・ルーティングの具体ルール

- **Viteビルド型**: 画像等は `src/` からの `import` か `public/` を使い、`base` にビルドが自動追従させる。`index.html` 内は Vite が `/client-demo-apps/<dir>/...` に解決する。**手書きの `/assets/...` を書かない。**
- **静的バンドル型**: `index.html` から `./app.js`・`./responsive.css` のように**相対パス**で参照する。
- **ルーティング**: React Router を使うなら Pages のサブパス直アクセスで 404 になり得る。`HashRouter` を使うか、直アクセスが要る画面は静的に成立させる。デモ用途では複雑なルーティングを避ける。
- 各デモの基本公開URLは必ず `/client-demo-apps/<demo-directory>/` の形。

## 9. ルート `index.html` の更新（デモ追加・削除で必須）

ルート `index.html` は「パスワードゲート付きのデモ一覧」。デモを**追加/削除したら必ず更新**する。

- 追加時: `.demo-list` の中に、既存カードと**同じ構造**で `.demo-card` を1枚足す:
  ```html
  <a class="demo-card" href="./<demo-directory>/">
    <div class="demo-info">
      <div class="demo-title"><!-- 日本語のデモ名 --></div>
      <div class="demo-slug"><demo-directory></div>
      <div class="demo-desc"><!-- 1〜2文のデモ概要 --></div>
    </div>
    <span class="demo-arrow">›</span>
  </a>
  ```
  - `href` は**相対 `./<dir>/`**（オーナー名をハードコードしない）。
- 削除時: 対応する `.demo-card` を削除する。
- ページ内のパスワードゲート（SHA-256 ハッシュ比較のスクリプト）は**触らない**。ハッシュ値・認証ロジックを変更依頼されていない限り、そのまま残す。

## 11. 判断に迷ったときのエスカレーション（正確な分岐）

**原則: 可逆で明確な作業は進める。不可逆・外部影響・仕様不明はユーザーに確認する。**

### 確認せず進めてよい（黙って実行し、結果を報告）

- 種別判定が明確なデモへの、指示範囲内のコード変更。
- `npm run build` の実行（Viteビルド型のビルドは許可された確認手段）。
- 指定作業ブランチへのコミット・push。
- ルート `index.html` のカード追加/削除（デモ追加・削除に伴う必須作業）。
- 明らかな架空データの追加・修正。

### 必ず `AskUserQuestion` で確認してから進める

- **デモの削除**、または**共有 `app.js` に影響し得る変更**（他デモに波及するため）。
- **`main` への push、または PR の作成**（明示依頼がない限りしない）。
- 静的バンドル型で「どうしても `app.js` 本体の改変が必要」に見えるとき（まずオーバーレイで解けないか検討し、無理なら確認）。
- ルート `index.html` の**認証ロジック・パスワードハッシュ**の変更。
- 既存の複数デモ／README／ルール文書にまたがる**大きめのリファクタ**。
- 指示が2通り以上に解釈できるとき（レビューコメントの意図が曖昧、等）。**推測で大きく動かない。**
- 実在情報かどうか判断がつかない固有名詞（社名・製品名等）をコミットに含めそうなとき。

### 手を止めて報告だけする（勝手に直さない）

- ビルドが**環境要因**（ネットワーク・依存の不整合）で失敗し、コード修正では直せないとき。
- 既存コミット済みコードに、今回の作業対象外の情報漏えい懸念を見つけたとき（勝手に改変せず、まず報告）。

### エスカレーション時に必ず添える情報

- 何をしようとして、どこで詰まったか（該当ファイル・行）。
- 選択肢と、各選択肢の影響（特に不可逆性・公開影響）。
- あなたの推奨案（あれば）。

## 13. 時短スキル（`.claude/skills/`）

このリポジトリの反復作業を安全・高速に行うためのスキルを用意しています。該当作業ではまずこれを使うこと。

| スキル | 使う場面 |
|--------|----------|
| `new-demo` | 新しいデモを追加する（種別・性格の判定込みで雛形生成＋ルートカード追加まで） |

| `uiux-audit` | デモの UI/UX を §5-UI・§5-UX・§2-2 の基準で監査する（違反箇所を該当ファイル・行つきで一覧化。「UIを完璧に」「UI確認して」等で起動） |
| `fix-demo` | 既存デモの修正を一気通貫で行う（種別判定 → 修正 → ビルド/整合 → preflight → コミット案まで） |
